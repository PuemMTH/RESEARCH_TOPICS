Judging LLM-as-a-Judge
with MT-Bench and Chatbot Arena

งานวิจัยแรกที่ศึกษาอย่างเป็นระบบว่า LLM แข็งแกร่งอย่าง GPT-4 สามารถทำหน้าที่เป็น "ผู้ตัดสิน" แทนมนุษย์ในการประเมินคุณภาพ chatbot ได้แม่นยำเพียงใด พร้อมเสนอ benchmark สองตัวและวิเคราะห์ bias สี่ประเภทของ LLM judge

Lianmin Zheng, Wei-Lin Chiang, Ying Sheng et al. NeurIPS 2023 · Datasets & Benchmarks UC Berkeley · Stanford · CMU arXiv:2306.05685
Abstract & Contributions

ภาพรวม

!

ปัญหาที่ Paper นี้แก้

Core Problem
  • Benchmark ดั้งเดิมใช้ไม่ได้กับ Chat Assistants: benchmark อย่าง MMLU, HellaSwag, HELM วัด core capability แต่ไม่สะท้อน human preference ในบทสนทนาจริง — โมเดลที่ได้คะแนน MMLU สูงไม่จำเป็นต้องถูก user ชื่นชอบ
  • Human Evaluation แพงและช้า: การให้มนุษย์ประเมินทุกคู่โมเดลไม่ scalable — ต้องการ automated surrogate ที่เชื่อถือได้
  • Traditional Metrics ล้มเหลวกับ Open-ended Questions: BLEU, ROUGE วัด string similarity ซึ่งไม่มีความหมายสำหรับคำตอบที่ถูกแต่เขียนต่างกัน
  • ยังไม่มี benchmark วัด Multi-turn & Instruction-following: ความสามารถที่สำคัญที่สุดของ chat assistant ยังไม่มี benchmark ที่เหมาะสม
★

Key Contributions

  • MT-Bench: 80 คำถาม multi-turn คุณภาพสูงใน 8 หมวด (Writing, Roleplay, Extraction, Reasoning, Math, Coding, STEM, Humanities) — ออกแบบมาให้แยกความแตกต่างของโมเดลได้ชัดเจน
  • Chatbot Arena: แพลตฟอร์ม crowdsourced battle ที่ผู้ใช้โหวตแบบ anonymous — เก็บ 30K votes จากการใช้งานจริง ไม่ใช่ pre-defined questions
  • Systematic Study of LLM-as-a-Judge: วิเคราะห์ bias 4 ประเภท + เสนอ solutions + ยืนยันด้วย 3K expert votes และ 3K crowdsourced votes ว่า GPT-4 judge ตรงกับมนุษย์ >80%
  • Hybrid Evaluation Framework: เสนอให้ใช้ capability benchmarks + preference benchmarks + LLM-as-a-judge ร่วมกัน
  • Public Datasets: เปิดเผย 80 MT-bench questions, 3K expert votes, 30K arena conversations
How It Works

Methodology

1

สองเส้นทางในการเก็บ Human Preference

Dual Benchmark Design

Paper ใช้สองแนวทางที่เสริมกัน: MT-Bench (controlled, expert evaluation) และ Chatbot Arena (wild, crowdsourced) ทั้งสองให้ ground truth สำหรับวัดว่า LLM judge แม่นยำแค่ไหน

A

MT-Bench

  • 80 คำถาม multi-turn ใน 8 หมวด (10 คำถาม/หมวด)
  • แต่ละคำถามมี 2 turns — turn แรกถามเนื้อหา, turn สองทดสอบ instruction-following (เช่น "rewrite เริ่มทุกประโยคด้วยตัว A")
  • ให้ 58 expert labelers (ระดับ grad student) ประเมิน ≥20 คู่ → ได้ ~3K votes
  • ทดสอบ 6 โมเดล: GPT-4, GPT-3.5, Claude-v1, Vicuna-13B, Alpaca-13B, LLaMA-13B
B

Chatbot Arena

  • แพลตฟอร์ม anonymous battle — user คุยกับ 2 โมเดล side-by-side แล้วโหวต
  • เก็บ 30K votes ใน 1 เดือน จาก 2,114 unique IPs
  • ครอบคลุม: GPT-4, GPT-3.5, Claude, Vicuna-7/13B, Koala-13B, Alpaca-13B, LLaMA-13B, Dolly-12B
  • ข้อได้เปรียบ: คำถามมาจากผู้ใช้จริง ไม่ได้ pre-defined → สะท้อน real-world use cases
2

LLM-as-a-Judge: 3 รูปแบบ

Judge Variations

Paper เสนอ 3 วิธีใช้ LLM เป็นผู้ตัดสิน ซึ่งสามารถใช้แยกกันหรือผสมกันได้ตามความเหมาะสม

  • Pairwise Comparison: ให้ LLM judge ดูคำตอบ 2 ชุดแล้วตัดสินว่าชุดไหนดีกว่าหรือ tie — แม่นยำกว่าแต่ไม่ scalable (O(n²) pairs)
  • Single Answer Grading: ให้คะแนนตอบเดียว — scalable กว่าแต่คะแนน absolute อาจไม่เสถียรถ้า judge model เปลี่ยน
  • Reference-guided Grading: ให้ reference answer ก่อนแล้วค่อยให้คะแนน — จำเป็นสำหรับคำถาม math/reasoning ที่ judge ต้องรู้คำตอบที่ถูก
3

4 Biases ของ LLM Judge และวิธีแก้

Bias 1: Position Bias

LLM judge ชอบคำตอบที่อยู่ ตำแหน่งแรก มากกว่า ทดสอบโดยส่งคำตอบที่เหมือนกัน 2 ชุด (เรียก GPT-3.5 ด้วย temperature 0.7 สองครั้ง) แล้วสลับตำแหน่ง

  • Claude-v1: consistent เพียง 23.8% (biased toward first 75%!)
  • GPT-3.5: consistent 46.2%
  • GPT-4: consistent 65% (ดีที่สุด แต่ยังมี bias)

Solution: เรียก judge สองครั้งด้วยตำแหน่งสลับกัน — declare win เฉพาะเมื่อชนะทั้งสองครั้ง ถ้า inconsistent ให้เป็น tie

Bias 2: Verbosity Bias

LLM judge ชอบคำตอบที่ ยาวกว่า แม้ข้อมูลซ้ำกัน ทดสอบด้วย "repetitive list attack" — เพิ่ม items ซ้ำเข้าไปในคำตอบเดิม

  • Claude-v1 & GPT-3.5: fail rate 91.3% (ถูกหลอกเกือบทั้งหมด)
  • GPT-4: fail rate เพียง 8.7%
Bias 3: Self-Enhancement Bias

LLM judge อาจชอบคำตอบของตัวเองมากกว่า เช่น GPT-4 favor ตัวเองด้วย win rate สูงกว่า human 10%, Claude-v1 สูงกว่า 25% — แต่ข้อมูลยังไม่พอสรุปได้ชัดเจน GPT-3.5 ไม่แสดง bias นี้

Bias 4: Limited Math/Reasoning Ability

LLM judge ถูกชักจูงโดยคำตอบที่ผิดในโจทย์ math/reasoning — แม้จะแก้ข้อนั้นได้ถ้าถามตรงๆ แต่เมื่อเห็นคำตอบผิดใน context ก็ตัดสินผิดตามไปด้วย

  • Default prompt: fail rate 14/20 (70%)
  • Chain-of-Thought prompt: 6/20 (30%)
  • Reference-guided: 3/20 (15%) ← ดีที่สุด
4

Agreement Metric

Definition

Agreement = ความน่าจะเป็นที่จะสุ่มเลือก 1 คน จากแต่ละกลุ่มแล้วทั้งสองเห็นตรงกันในคำถามที่สุ่มมา (ไม่ใช่คนเดียวกัน) วิธีนี้ทำให้เปรียบเทียบ human-human agreement กับ GPT4-human agreement ได้อย่างยุติธรรม

ทดสอบทั้ง Setup S1 (รวม tie) และ S2 (ไม่รวม tie) — S2 ให้ค่า agreement สูงกว่าและมีความหมายทางการประเมินมากกว่า

Key Takeaways
  • GPT-4 judge ตรงกับมนุษย์ 85% ซึ่งเท่ากับหรือสูงกว่า human-human agreement (81%) — หมายความว่า GPT-4 consistent กว่ามนุษย์เฉลี่ยด้วยซ้ำ

  • GPT-4 single-answer grading ทำงานได้ใกล้เคียง pairwise แต่ scalable กว่า — เหมาะสำหรับ large-scale evaluation

Gaps & Future Work

ข้อจำกัดและทิศทางต่อไป

Limitations ที่ Paper ยอมรับ
  • เน้น helpfulness เท่านั้น: ไม่ได้ประเมิน safety, honesty, harmlessness — ซึ่งสำคัญมากในการ deploy จริง
  • Metric เดียวสำหรับทุกมิติ: helpfulness มีหลาย dimension (accuracy, relevance, creativity) แต่ถูก collapse เป็นคะแนนเดียว ไม่มีการแยกวิเคราะห์
  • Self-enhancement bias ยังสรุปไม่ได้: ข้อมูลยังไม่พอ และทำ controlled study ยากเพราะต้อง rephrase คำตอบให้เหมือน style ของโมเดลอื่น
  • Solutions ยัง preliminary: position swapping และ reference-guided judge แก้ได้บางส่วน แต่ไม่ครบถ้วน — ต้องการ advanced methods เพิ่มเติม
  • Multi-turn prompt complexity: การแสดง 2 full conversations ใน single prompt ดีกว่า 2 prompts แยก แต่ทำให้ prompt ยาวมากและ expensive
Future Directions ที่ Paper ระบุ
  • Benchmarking chatbots at scale ด้วย broader categories
  • Open-source LLM judge ที่ align กับ human preference (ไม่ต้องพึ่ง GPT-4)
  • Enhancing open models' math/reasoning capability
Connection to My Research

ความเกี่ยวข้องกับวิทยานิพนธ์

What I Borrow

LLM-as-a-Judge สำหรับ Figure Caption Evaluation — แทนที่จะใช้ BLEU/METEOR (ซึ่ง paper นี้เองก็วิจารณ์ว่าไม่เหมาะกับ open-ended questions), วิทยานิพนธ์ควรใช้ GPT-4 judge ประเมินคุณภาพ caption ที่ generate ออกมา

Reference-guided grading — เมื่อประเมิน caption ของ scientific figure, สามารถให้ original caption จาก paper เป็น reference เพื่อให้ judge รู้ว่า "คำตอบที่ถูก" คืออะไร ลด fail rate จาก 70% → 15%

Position swapping technique — ถ้าต้องทำ pairwise comparison ระหว่าง caption จากหลายวิธี ต้องสลับตำแหน่งทุกครั้งเพื่อหักล้าง position bias

Where I Diverge

Domain-specific evaluation — scientific figure captions มี structure ที่เฉพาะเจาะจงกว่า chatbot responses ทั่วไป เช่น ต้องอธิบาย x-axis, y-axis, trend, และ key finding — จะต้อง customize prompt ให้เหมาะกับ scientific domain

Multi-dimensional scoring — ต่างจาก paper นี้ที่ใช้ single score, งาน caption evaluation ควรแยก dimension เช่น factual accuracy, visual grounding, completeness, fluency

RAG context as reference — ในระบบ RAG ที่ดึงบริบทจาก paper มาช่วย generate caption, judge ควรได้ดูทั้ง retrieved context และ original caption เพื่อประเมินว่า RAG ช่วยเพิ่มคุณภาพจริงหรือไม่

Open Questions

GPT-4 verbosity bias กับ caption: caption ที่ยาวกว่าจะได้คะแนนสูงกว่าเสมอไหม ทั้งที่ caption สั้นๆ ที่ตรงประเด็นอาจดีกว่า? ต้องออกแบบ evaluation prompt ที่ควบคุม verbosity bias ด้วย

Self-enhancement ใน scientific domain: ถ้าใช้ GPT-4 ทั้ง generate และ evaluate captions, self-enhancement bias จะส่งผลมากแค่ไหน? ควรใช้ judge โมเดลอื่น (Claude, Gemini) เป็น cross-check

Open-source judge: Paper เสนอ fine-tuning Vicuna-13B เป็น judge — น่าสนใจว่า fine-tuning เฉพาะ domain scientific figures จะได้ judge ที่ดีแค่ไหนโดยไม่ต้องพึ่ง GPT-4 ซึ่งแพงมาก
